## システム設計詳細検討項目

### 1. 義体プログラミング言語 (Core Prosthetic Body Programming Language)

*   **NC工作機械のGコード/Mコードに倣った、テキストベースのコマンドプログラミング**
    *   **`mv`コード (Movement Code - Gコード相当):** 義体の移動を指示する。
        *   **`G00`相当 (高速・非精密移動):** `mv x[数値] y[数値] z[数値] FAST` - 可能な限り最速で移動するが、途中経路は考慮しない。衝突のリスクあり。
        *   **`G01`相当 (精密・直線移動):** `mv x[数値] y[数値] z[数値] SPEED [数値]` - 指定された速度で直線的に目標地点へ移動する。経路の安全性や精密な動作を重視。
    *   **`mo`コード (Motion/Action Code - Mコード相当):** 義体のアクションやシステム的な処理を指示する。
        *   **モーダルコード:** 一度発動すると、他のコードが上書きされない限りそのモード（例: 接客モード）が継続する。
        *   **`M30`相当 (プログラム終了/義体待機):** `mo END_PROGRAM` - 現在のプログラムの実行を終了し、義体を待機状態にする。
        *   **`M06`相当 (義体状態変化/ツールチェンジ):** `mo CHANGE_EQUIPMENT [装備ID]` - 義体の装備を変更する。パブであれば「衣装チェンジ」、戦場であれば「武器変更」など。
*   **コマンドセットの例:**
    *   **`mv`コード:**
        *   `mv X[数値] Y[数値] Z[数値]`: 絶対座標または相対座標での移動。
        *   `mv SPEED [数値]`: 移動速度の設定。
        *   `mv TO_TARGET [ターゲットID/タイプ]`: 特定のターゲット（顧客、敵、オブジェクト）への移動。
    *   **`mo`コード:**
        *   `mo [番号] [アクション名]`: 定義済みモーションの実行。
            *   例: `mo01 SHOOTING` (射撃)、`mo02 RELOAD` (リロード)、`mo03 GREET_CUSTOMER` (顧客に挨拶)、`mo04 SERVE_DRINK` (ドリンク提供)
        *   `mo [番号] [アクション名] PARAM [数値/文字列]`: パラメータ付きモーションの実行。
            *   例: `mo05 SMILE_LEVEL 3` (笑顔レベル3で笑顔)、`mo06 USE_SKILL [スキル名]`
*   **パラメータ:** 各コマンドには、NC工作機械のように数値パラメータ（移動量、速度、角度など）や、文字列パラメータを与える。
*   **`mo`コードのシーケンス例 (シェルスクリプトのようなブロック実行):**
    *   例: `mo02 1 s01 s02 2 end`
        *   これは`mo02`という「接客モード」を開始し、`s01`というサブアクションを1回実行後、`s02`というサブアクションを2回実行し、`end`でこのシーケンスを終了するイメージ。
        *   `s01` (例: 横に座る)
        *   `s02` (例: お酌する)
    *   このように、一連の動作を「ブロック」として定義し、終了 (`end`) する。

### 2. プログラムの作成と管理 (Program Creation and Management)

*   **インターフェース:** ゲーム内でプレイヤーはどのように「コード」を作成・編集しますか？
    *   **シェルスクリプトのようなテキストエディタ形式:** コマンドとパラメータを手打ちするシンプルなテキストエディタ画面を想定。
    *   視覚的にブロックを繋いでいくような**ビジュアルスクリプティング**方式ですか？
    *   以前ドクターひろしが言及された**GTPコードやYコード**のような独自の数値コード体系をゲーム内のプログラミング言語として採用するイメージですか？
*   **プログラム管理:** 作成したプログラムは、義体ごとに保存・ロードできる「プログラムライブラリ」のような形で管理しますか？
*   **事前プログラミング:**
    *   **顧客情報に基づいた事前プログラミング:** パブでの接客では、お客様の情報を事前に分析し、最適な行動をプログラミングしておくことが求められる。
    *   これは、NC工作機械で加工対象物の情報からプログラムを組むことに類似。

### 3. プログラムの実行とフィードバック、デバッグ (Execution, Feedback, Debugging)

*   **実行:** 作成したプログラムは、義体に「アップロード」され、プレイヤーの指示（例: 「プログラム実行」ボタン）で義体がそのシーケンスを自動で実行するイメージですか？
*   **プログラム実行中の変更不可:** 基本的に、接客や戦闘が始まったら義体のプログラムは変更できない。事前プログラミングの重要性が高い。
*   **エラー検出:** プログラムが失敗した場合（例: 誤ったコマンド、環境との不整合、リソース不足）、ゲーム内でどのようなフィードバックがありますか？（例: エラーログ、義体の挙動の可視化、デバッグモード）
*   **最適化:** より効率的、あるいは顧客満足度が高い、または戦闘で有利なプログラムを作成することで、報酬や評価が変わるような要素はありますか？

### 4. 二つの主要なゲームループ (Pub & Battlefield Loops) の連携 (Game Loop Integration)

*   **相互作用:** 「義体パブ」と「戦場での義体兵士操作」という二つの要素はどのように繋がり、プレイヤーの体験を豊かにしますか？
    *   パブでの収益が戦場での義体の修理・強化費用になる？
    *   戦場での功績がパブの評判を上げ、新しい客層や義体、ミニゲームをアンロックする？
    *   戦場での義体の損傷がパブの稼働に影響する？
*   **プレイヤーの役割:** プレイヤー自身は、義体とは別の存在（例: 脳だけ、AI、生身の人間）として、複数の義体を切り替えながら操作するイメージですか？
*   **代替義体の出動:** プログラム実行中に状況が悪化した場合、別の義体を投入することで状況に対応する戦略が可能。これはNCの「ツールチェンジ」や「ワークチェンジ」のような概念に近い。

### 5. 世界観「ネオワールドのアーカイブ世界」のシステムへの影響 (Lore Integration)

*   **アーカイブならではの要素:**
    *   義体の修理や改造が「データの復元」「コードの書き換え」のような表現になる？
    *   戦闘での死亡や義体破壊が「データロスト」や「ロールバック」のような表現になる？
    *   アーカイブの「バグ」や「ノイズ」が敵やクエスト、システムに影響を与える？
